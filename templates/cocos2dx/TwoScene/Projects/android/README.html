Building Android README

Author's Note: The following gives a detailed overview for everything needed in order to compile a RapidGame project for Android. As such, there will be redundant information for those who already have experience with Android development. If that is the case, it is still recommended that you read through the entire document, as there is important information specific to RapidGame that you should know about so that you can properly configure your project to fit your needs. - Samuel Ørsnæs

TL;DR

  • Make sure you have all the programs that are under Prequisite Downloads
  • Check your environment variables against the list in Setup Environment Variables
  • cd [APPNAME]/Projects/android && make
  • Plug in your Android device via USB
  • make run
  • Find and launch app

You can create, delete, and edit source files under [APPNAME]/Projects and assets under [APPNAME]/Assets as you wish. Changes in these folders will be included in the Android build automatically.

Prerequisite Downloads

  • Android Studio (includes Android SDK and many other tools)
  • Android NDK (C/C++ development for Android)
  • Apache Ant (needed to build .apk's)
  • Python (2.7.x preferred, although 3.4.x should be fine)
  • Cygwin (for Windows only; download the "Devel" packages as well in setup)

As the Android NDK and Ant do not come with installers but are compressed folders, you choose yourself where you want to unpack the folder. According to the NDK, the path to it may not contain any spaces (e.g. /path/to my/ndk is not allowed).

After downloading Android Studio, you will need to download the APIs for the OS version that you want to target. You could for example set Android 4.3.1 (API 18) as your target OS. It turns out that the template project that comes with RapidGame targets API 18, while cocos2d-x itself targets API 10. This means that you must download these particular APIs before you can build the template project. In order to do so, you go to Android Studio -> Configure -> SDK Manager -> [Select and install appropriate packages]. If you had previously created a project in Android Studio, then launching Android Studio will open up directly to the IDE. To navigate to the SDK Manager from there, you go to Tools -> Android -> SDK Manager.

(Windows only) Cygwin must be installed for all users (the recommended option in Cygwin's setup), which installs Cygwin in your root directory (C:\Cygwin or C:\Cygwin64). You should of course install the 64-bit version if you have a 64-bit machine. After you download the setup program found on their front page, you should keep the file stored somewhere safe in order to update your installation and install more programs as you become comfortable with it. The setup program itself may seem a bit intimidating, so just as a quick guide, here's what you do: Launch setup -> Next -> Next (Install from Internet) -> Next (Install for all users) -> Choose a directory that you won't delete on a whim -> Next -> Next (Direct connection) -> Next (Any mirror works) -> "Devel" change from `Default` to `Install` -> Next (Option `Select required packages` should be checked) -> [Wait] -> Finish.

(Windows only) In addition, although certain commands can be run outside of Cygwin, it is recommended that you run all of the possible commands listed in this README in a Cygwin shell for ease of use. You must additionally run the shell as administrator each time you want to create a new project with rapidgame create in order to properly create the lib symlink. The first time you run rapidgame create, which will automatically prebuild the libraries for you as well, if you did not use an admin Cygwin shell, then the libraries will have been improperly placed and you will need to delete rapidgame/[VERSION] and rapidgame/latest, the latter of which will have been improperly created as a folder instead of a symlink. After that, delete the defective project that was created by rapidgame create and run the command again in a proper admin Cygwin shell, and the project and libraries will be properly created.

Setup Environment Variables

Before building your first project, you should make sure all the necessary environment variables are set. It is assumed you already know how to work with environment variables for your system, but if you need help, refer to the bottom of this README.

ANDROID_SDK_ROOT = /path/to/sdk
ANT_HOME = /path/to/ant
ANT_ROOT = /path/to/ant/bin
JAVA_HOME = /path/to/jdk
NDK_ROOT = /path/to/ndk
PATH = $PATH,
       /path/to/python,
       $JAVA_HOME/bin,
       $NDK_ROOT,
       $ANDROID_SDK_ROOT/build-tools/[VERSION],
       $ANDROID_SDK_ROOT/platform-tools,
       $ANDROID_SDK_ROOT/tools

The above is a generic representation of the environment variables and not proper syntax. It is for ease of readability and up to you to translate the information for your system. Because RapidGame downloads and configures cocos2d-x and cocos2d-js for you, there is no need to set any environment variables related to Cocos.

(Windows only) In addition to setting the above environment variables for your host OS, you must also set ANDROID_SDK_ROOT and NDK_ROOT in Cygwin's ~/.bash_profile as you would on Mac. The formatting should be as follows: export ANDROID_SDK_ROOT=C:/path/to/sdk, and you follow that pattern for NDK_ROOT as well. See the link to `Mac` if you are confused.

Windows defaults:

  • Android SDK is installed in C:\Users\[USERNAME]\AppData\Local\Android\sdk
  • JDK is installed in C:\Program Files\Java\jdk1.X
  • Python is installed in C:\PythonXX

Building Your Project

MyGame = example root project directory

Anytime you actually build an .apk file (any of the first 3 commands listed below), make will automatically search in MyGame/Projects for all your C and C++ source files (.c, .cpp, .cc, .cxx) and update the changes you've made (whether creating or deleting files) for the build list. As header files aren't source files, you don't need to worry about them. Other filetypes such as .m and .mm will be ignored for the Android build. You can edit MyGame/Projects/android/makeignore and add the C and C++ files you do not want included in the Android build, one file per line. After editing makeignore and running make [target], your project will be built according to the file changes in MyGame/Projects and the changes in makeignore. Your C/C++ filenames may not contain any spaces in them.

It's assumed that the RapidGame libraries have been built for Android by this point and you've created your project using rapidgame create cocos2dx or rapidgame create cocos2djs. You are able to edit your source files in whatever IDE/text editor you like, but you should save the files in MyGame/Projects. For assets, add them to MyGame/Assets and they will automatically be included in your Android game under MyGame/Projects/android/assets/Assets when you run make. The assets directory and its subdirectories are destroyed upon execution of make clean. When you are ready to build your game, you will have several options. You will first open Terminal on Mac or a Cygwin shell on Windows and change the directory (cd) to MyGame/Projects/android. The following is a list of commands you may want to execute after that step (refer to MyGame/Projects/android/makefile if you want to make any modifications):

  1. make: Build the debug version of your game and sign the .apk with the debug key. If you have never signed a debug Android application before and are prompted to enter information pertaining to the debug key, here is the info you need: 

    Keystore name: debug.keystore (absolute path: ~/.android/debug.keystore on Mac and C:/Users/[USERNAME]/.android/debug.keystore on Windows)
Keystore password: android
Key alias: androiddebugkey
Key password: android
CN: CN=Android Debug,0=Android,C=US
  2. make release: Build the release version of your game but still create the .apk file with the debug key.
  3. make distribution: Build the release version of your game and sign the .apk with the release key. For info on how to build a release key and sign your app, refer to this page. If you prefer to do everything on the command line, take a look here. The commands are OS-agnostic, so they are the same on Mac and Windows.
  4. make run: Install your game on your Android device and begin printing log output to the console. Note that you must have your device plugged in via USB before running this command. In addition, make run does not launch your app. That must be done by you. If you get a "waiting for device" message after running this command, press ^C to end the command's execution and run adb devices to check if your device has been recognized yet. As for the log output, the default settings in the makefile are to print debug output from your app and only fatal errors from everywhere else. For more about how logging works, refer to this page and feel free to modify the makefile to your needs. If you want all future projects to keep a particular setting, you can modify rapidgame/templates/cocos2dx/TwoScene (replace cocos2dx with cocos2djs if need-be) or create your own templates.
  5. make run-relase: Functionally equivalent to make run by default, as it installs APPNAME-debug.apk on your device and logs output to the console, exactly as make run does. However, you can modify the makefile to change what kind of logs you will see when you execute make run versus when you execute make run-release.
  6. make run-distribution: Installs APPNAME-release-aligned.apk on your device and logs output to the console.
  7. make clean: Remove all the files and folders that were created when you built your game.
  8. make start-emulator: You will first need to create an Android Virtual Device (AVD) using AVD Manager, which can be found at $ANDROID_SDK_ROOT/AVD Manager, or by opening Android Studio and navigating to the Android Tools and selecting it from there. The AVD you create should be set to use the ARM architecture, because 1) RapidGame Android projects can only build for ARM, and 2) x86 requires you to download and configure several things before you can emulate that architecture. You will then need to uncomment #emulator -avd [device_name] in the makefile by removing the # and then replace [device_name] with the name of the AVD you want to use for testing. In addition, uncomment the lines under run-emulator. Although the emulator is useful for testing your game in different environments, you should thoroughly test your game on at least a few actual Android devices before you release it.
  9. make run-emulator: Installs the app on your chosen AVD. You must have the emulator running, and you must execute this command in a new shell if you had run make start-emulator, as the latter command must continue running until you are finished.

Modifying for Your Needs

If you are new to Android development, you may have questions about all the files that are in MyGame/Projects/android. First off, these files do not need to be changed in any way in order to get the template project running. However, they are of course used in some way to specify the metadata of your game, such as what the target Android OS version is, or whether your game starts in portrait or landscape, or what kind of permissions you want access to and what kind of hardware features your game requires. The following is a summary of all the files you may want to modify for some reason or another in order to fit your needs, meaning not all of the files will be mentioned here. If you still want to modify a specific feature and don't know how to do it, this list is meant to help you with Googling your problem to find the answer. However, this page is a good starting point.

  • .identifier: The package identifier for your game.
  • .name: The name of your game.
  • AndroidManifest.xml: This is the main file you will need to muck around with if you want to change any metadata about your game. Refer to the page mentioned above to see what's contained in the file itself and how to modify it.
  • ant.properties: Add your keystore information to this file so that you are automatically prompted for the release key's password when you run make distribution. Refer to the info mentioned under Building Your Project if you don't know what a keystore is or how to make one.
  • build_files.py: Makes sure the list of source files is updated everytime you run make. Should not be changed unless you know what you're doing.
  • build_native.py: The script that actually compiles your files and puts them in a shared library before ant is used to pack it in an .apk. Should not be changed in any way.
  • makefile: Tells make what to do to build your project for each of the possible targets. You could for example change how verbose you want logcat to be when printing output to the console.
  • makeignore: The list of C/C++ source files under MyGame/Projects that you don't want included in the Android build of your game.

Building for Different Architectures

As your app is using native code (C++), your final .apk will contain code compiled for a specific architecture. If you were writing a pure Java app, that app would run on any device without needing to compile for each architecture thanks to the JVM. By default, your Android game is set to build for the ARM (armeabi) architecture, which a huge majority of Android phones use (think equivalent to Intel's dominance in desktop computing). RapidGame has the ability to prebuild the Android cocos2d-x libraries for ARM, ARMv7, and x86, but unfortunately trying to build the template project after setting the architecture to x86 will cause the build to fail, so the Android-x86 libraries are skipped over in rapidgame prebuild to save time. Even so, the x86 devices that have libhoudini on them are able to run ARM apps, albeit more slowly. In any case, the two currently supported architectures you can build for are ARM and ARMv7. ARMv7 is backwards compatitable with ARM, so you can just build your app for ARM and it will run on newer ARMv7 devices. The main advantage of ARMv7 is faster floating point operations, so you may find your game performs better on newer devices. For more specific info on the different architectures, refer here. You can find out much more extensive documentation about the NDK in general by going to $NDK_ROOT/docs/Start_Here.html.

In order to actually get your project to compile for both ARM and ARMv7, you can edit APP_ABI in MyGame/Projects/android/Application.mk to APP_ABI := armeabi armeabi-v7a. When you rebuild your project with make, the .apk will now contain the files for both ARM and ARMv7, and when it is installed on a device, the necessary files will be used depending on the processor. One thing that must be noted is that when you build for ARMv7, you will get a long list of warning messages, and these messages will continue scrolling on the screen for several minutes until finally your game finishes building. There is nothing that can be done to prevent these warning messages (if you discover a way, please let us know on RapidGame's issues page (requires a Github account)), but they are harmless. Your game will still compile and run without issue. However, to speed up build time, you should only build for ARM as you build and test the core of your game. If your .apk gets to be over 50MB and you are considering submitting multiple .apk's for your game, then there are certain rules you must be aware of.

Environment Variables Intro

If you're not familiar with environment variables, they are variables that are accessible by any program on your computer, which makes them very useful and important. One example of this usefulness is that when you type a command like python in your console, your computer doesn't automatically know where to find it. However, all major OS's have a variable called PATH, that is a list of directories, which your computer searches in to find if there's something related to what you typed on the command line. So when you type python in your shell, the OS looks in all the directories and when it finds a program named python, it executes that program. The following will explain how to properly set environment variables for your particular OS.

Mac

Your environment variables can be set in ~/.bash_profile (a plain text file). You will find other things written in the file already, which you should ignore. Go to the bottom of the file to add your own stuff. The format is as follows:

export var1=/path/to/var1
export var2=/path/to/var2
# We want to keep the original variable and add to it
export PATH=$PATH:/path/to/something:/path/to/another/thing

When you are done editing the file, save and close it and then run source ~/.bash_profile and you're good to go. If you want to double-check, you can type echo $VAR, which will print the value of VAR back to you.

As a brief additional explanation if you're unfamiliar with the command line, ~ is your home directory and can be used in your terminal to get you back home (cd ~). After changing to your home directory, you can type pwd to find out where "home" actually is. Another interesting point is dotfiles. These are files that start with a . and cannot be seen by typing ls. You can type ls -a to see all of the files in a directory, including dotfiles.

Windows

For Windows, the easiest way to edit your environment variables is by pressing the Windows button, searching "environment variables", and clicking on the link that comes up. A window should open that is divided into user and system variables, and we are only interested in editing user variables. Be sure to check if the variable already exists before trying to create a new one. Each variable has a name and a value, so if we were setting ANDROID_SDK_ROOT, then that would be the name, and the value might be C:\Users\John\AppData\Local\Android\sdk. If you want to add to an already existing variable, then you add the path to add to the beginning of the value and then place a ; between that and the rest of the value, e.g. we want to add Python's path to some other paths, so the final result would look like: C:\Python27;C:\cygwin64\bin;C:\other\things.
